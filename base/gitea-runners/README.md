# Gitea Runners

## What it does

Gitea Actions Runners are self-hosted CI/CD agents that execute Gitea Actions workflows. They connect to the Gitea instance in this cluster, poll for queued workflow jobs, and run them inside containers on cluster nodes.

The runner is deployed as a Kubernetes `Deployment` and uses a service account with minimal permissions. It mounts a Docker socket (or uses a Kubernetes executor, depending on configuration) to run job containers.

## Why it was added

The cluster hosts its own Gitea instance. Having self-hosted runners means CI/CD pipelines (build, test, deploy) can run inside the cluster with access to internal resources — without requiring outbound connectivity to external runner services.

## Dependencies

- **gitea** — runners authenticate to the Gitea instance using a runner registration token (`giteaRunnerTokenBootstrap.yaml`). Gitea must be running and healthy before runners can register.

## Dependents

None — no other components in this repository depend on Gitea runners.

## User Guide

### Checking runner registration status

In the Gitea UI: **Site Administration → Runners** (or per-repository **Settings → Runners**).

From the cluster:

```bash
kubectl -n <gitea-runners-namespace> get pods
kubectl -n <gitea-runners-namespace> logs -l app=gitea-runner --tail=50
```

### Runner token rotation

The runner registration token is stored in a Kubernetes `Secret` generated by `giteaRunnerTokenBootstrap.yaml`. If the token needs to be rotated:

1. Generate a new runner token in the Gitea UI.
2. Update the `Secret` in the cluster.
3. The runner pod will restart automatically via Reloader.

### Runner labels and job routing

Gitea Actions workflows can target specific runners using the `runs-on` field:

```yaml
jobs:
  build:
    runs-on: [self-hosted, linux]
```

Runner labels are configured in `giteaRunnerConfigMaps.yaml`.
